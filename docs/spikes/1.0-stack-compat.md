## Copy-pasteable Gradle snippets

### Top-level `build.gradle.kts` — KGP classpath upgrade

AGP 9 bundles its own Kotlin version. To override it with 2.3.20 (required for KSP 2.3.9
and Metro 1.1.1), add a `buildscript` block **before** the `plugins` block:

```kotlin
// Force Kotlin 2.3.20 — AGP 9 bundles an older KGP; this classpath entry wins.
buildscript {
  dependencies {
    classpath("org.jetbrains.kotlin:kotlin-gradle-plugin:2.3.20")
  }
}

plugins {
  id("com.android.application") version "9.0.1" apply false
  id("dev.zacsweers.metro")     version "1.1.1"  apply false
  id("com.google.devtools.ksp") version "2.3.9"  apply false
}
```

## Metro × Decompose pattern (from POC)

### `AppGraph.kt` — root DI graph

```kotlin
package com.example.spike

import dev.zacsweers.metro.DependencyGraph
import dev.zacsweers.metro.Provides

@DependencyGraph
interface AppGraph {
  val eventRepository: EventRepository

  @Provides
  fun provideEventRepository(): EventRepository = DefaultEventRepository()
}
```

`@DependencyGraph` triggers Metro's compiler plugin. `createGraph<AppGraph>()` at call site
instantiates the generated implementation.

### `RootComponent.kt` — Decompose root component, Metro-injected dependency

```kotlin
package com.example.spike

import com.arkivanov.decompose.ComponentContext
import com.arkivanov.decompose.childContext

class RootComponent(
  componentContext: ComponentContext,
  private val eventRepository: EventRepository,
) : ComponentContext by componentContext {

  fun createEventDetailComponent(): EventDetailComponent =
    EventDetailComponent(
      componentContext = childContext("event_detail"),
      eventName = eventRepository.getEventName(),
    )
}
```

Key points:
- `ComponentContext by componentContext` — standard Decompose delegation, no framework coupling.
- `eventRepository` arrives via constructor; the Metro graph is assembled in `MainActivity`
  and passed in — no framework annotation needed on the component itself.
- `childContext("event_detail")` creates a scoped child lifecycle automatically.

### `EventDetailComponent.kt` — child component

```kotlin
package com.example.spike

import com.arkivanov.decompose.ComponentContext

class EventDetailComponent(
  componentContext: ComponentContext,
  val eventName: String,
) : ComponentContext by componentContext
```

### `MainActivity.kt` — wiring site

```kotlin
val graph = createGraph<AppGraph>()

val root = RootComponent(
  componentContext = defaultComponentContext(),
  eventRepository = graph.eventRepository,
)

val detail = root.createEventDetailComponent()
```

`defaultComponentContext()` is the standard Decompose Activity extension. The Metro graph is
created once here and its values passed down as plain constructor arguments — clean separation
between DI graph lifecycle and Decompose component lifecycle.

## Gotchas

### AGP 9 built-in Kotlin / KGP classpath upgrade — most important

AGP 9.0.x ships with a bundled Kotlin Gradle Plugin. If you apply `kotlin-android` or any
Kotlin plugin via the `plugins {}` block without an explicit version AND without the classpath
override, you may silently pick up AGP's bundled KGP (e.g. 2.1.x) instead of 2.3.20. This
causes a version mismatch with KSP `2.3.9` (which requires Kotlin 2.3.x) and Metro 1.1.1.

Fix: add the `buildscript { dependencies { classpath("org.jetbrains.kotlin:kotlin-gradle-plugin:2.3.20") } }`
block to the **root** `build.gradle.kts` (see snippet above). The classpath entry takes
precedence over AGP's bundled KGP.

Symptom when missing: KSP reports "kotlin version mismatch" or Metro's IR plugin fails to
load with an incompatible API error.

### KSP version naming

KSP versions follow `<kotlin-version>-<ksp-patch>` in older releases but since Kotlin 2.x the
published artifact is just `2.3.9` (no patch suffix). Verify the correct coordinate exists on
Maven Central before pinning: `com.google.devtools.ksp:symbol-processing-api:2.3.9`.

### Metro `@DependencyGraph` on interface, not class

Metro requires the DI graph declaration to be an `interface` (or abstract class). A concrete
`class` annotated with `@DependencyGraph` will not compile — the code-gen target must be
abstract so Metro can generate an implementation.

### Decompose `defaultComponentContext()` requires `ComponentActivity`

`defaultComponentContext()` is an extension on `ComponentActivity`, not on the base `Activity`.
The dependency `androidx.activity:activity:1.10.1` (or the `-ktx` variant) must be on the
classpath. Without it the Activity file fails to resolve `defaultComponentContext`.

### No `kotlin-android` plugin needed with Metro on a pure DI module

If a module only contains Metro graph interfaces and plain Kotlin (no Android resources, no
Compose), it can use `id("org.jetbrains.kotlin.jvm")` instead of `kotlin-android`, reducing
configuration overhead.
